Custom TWRP recovery build for the **LG K8+** (model `LM-X120BMW`) powered by the **MediaTek MT6737M** chipset (32-bit ARM, Cortex-A53 in 32-bit mode).

---

## Overview

This project provides a complete, CI-driven TWRP build pipeline with **runtime hardware auto-discovery**. The GitHub Actions runner queries a connected device via ADB to extract framebuffer resolution, brightness path, USB UDC controller name, and other hardware parameters, then injects them dynamically into the build.

### Problem History

Previous builds (#1–#7) booted (splash screen visible) but exhibited two critical failures:

1. **TWRP UI did not render buttons** — the framebuffer was accessible but `/sbin/recovery` crashed silently when initializing the UI.
2. **ADB did not appear** — no USB gadget was bound, so no debugging was possible.

**Root cause (build #7 ramdisk analysis):**

- `init.rc` imported `/init.recovery.${ro.hardware}.rc` conditionally.
- `ro.hardware` = `mmh6lm` (LG codename), but the actual file in the ramdisk was named `init.recovery.mt6739.rc` (SoC name).
- The import silently failed → no USB/ADB setup → no framebuffer permissions → UI crash.
- Additionally, `init.recovery.mt6739.rc` contained `rm /config/usb_gadget/g1/configs/b.1/f2` which broke the gadget bind, and hardcoded `write /config/usb_gadget/g1/UDC "musb-hdrc"` which failed on revisions using `musb-hdrc.0` or `dwc3.0`.

### Solution

1. **Hardcoded SoC import** in `init.rc` — imports `init.recovery.mt6739.rc` directly, with a fallback to `${ro.hardware}`.
2. **Dynamic UDC** — reads `/sys/class/udc/` at runtime instead of hardcoding the controller name.
3. **Removed `rm f2`** — the line that broke ADB gadget binding is gone.
4. **`ro.secure 0`** — set in both `init.rc` and `init.recovery.mt6739.rc` to allow ADB in recovery.
5. **Auto-discovery** — `scripts/discover_device.sh` queries the device via ADB and generates `device_vars.mk` with real hardware parameters.
6. **Fallback** — `scripts/fallback_vars.sh` provides estimated values if no device is connected, so the build never blocks.

## GitHub Secrets Setup

The pipeline supports multiple connection modes. Configure secrets in **Settings → Secrets and variables → Actions**.

### Option 1: ADB over TCP (recommended for networked devices)

| Secret | Value | Example |
|--------|-------|---------|
| `ADB_IP` | IP address of the device on your network | `192.168.1.100` |
| `ADB_PORT` | ADB TCP port | `5555` |

**To enable ADB over TCP on the device:**

```bash
# On a computer with USB connection to the device:
adb tcpip 5555

# Then disconnect USB and connect via network:
adb connect 192.168.1.100:5555
```

> Requires USB debugging enabled and (for `tcpip` command) either root or developer options access.

### Option 2: SSH_PRIVATE_KEY (remote self-hosted runner)

Use this when the device is physically connected via USB to a remote machine:

1. Generate an SSH keypair and add the public key to the remote runner's `~/.ssh/authorized_keys`.
2. Store the private key in the `SSH_PRIVATE_KEY` secret.
3. In the workflow, use `appleboy/ssh-action` or a custom step to SSH into the remote machine and run `adb` commands there.

```yaml
# Example (not included in default workflow):
- name: Discover via SSH
  uses: appleboy/ssh-action@v1
  with:
    host: ${{ secrets.REMOTE_HOST }}
    key: ${{ secrets.SSH_PRIVATE_KEY }}
    script: |
      adb devices
      bash /path/to/scripts/discover_device.sh
      scp device_vars.mk device_vars.env runner:/workspace/
```

### Option 3: Self-hosted runner with USB device (best for continuous CI)

1. Set up a [self-hosted GitHub Actions runner](https://docs.github.com/en/actions/hosting-your-own-runners) on a machine with the device connected via USB.
2. Add the label `android-device` to the runner.
3. Change `runs-on` in the workflow:

```yaml
jobs:
  build:
    runs-on: [self-hosted, android-device]
```

4. The device must have USB debugging enabled and be authorized on the runner.

### Option 4: No device (fallback mode)

If no secrets are configured and no device is detected, the pipeline automatically runs `scripts/fallback_vars.sh` with hardcoded MT6737M defaults. The build succeeds but uses estimated values — verify on a real device before flashing.

## How Auto-Discovery Works

```
┌─────────────────┐     ┌──────────────────────┐     ┌─────────────────┐
│  GitHub Runner  │────▶│  scripts/            │────▶│  device_vars.mk │
│  (Ubuntu 22.04) │     │  discover_device.sh   │     │  device_vars.env│
└─────────────────┘     └──────────────────────┘     └────────┬────────┘
                               │                              │
                               │ adb shell getprop            │ -include
                               │ adb shell cat fb0            ▼
                               │ adb shell find leds/    ┌─────────────┐
                               │ adb shell ls udc/       │ BoardConfig │
                               │                         │ .mk         │
                               ▼                         └─────────────┘
                        ┌──────────────┐
                        │  Fallback?   │──YES──▶ scripts/fallback_vars.sh
                        │  (no device) │
                        └──────────────┘
```

1. **`discover_device.sh`** connects via ADB and extracts:
   - `ro.hardware`, `ro.product.device`, `ro.serialno`, `ro.boot.mode`
   - Screen resolution from `/sys/class/graphics/fb0/virtual_size` (fallback: `dumpsys display`)
   - Brightness path from `/sys/class/leds/*/brightness`
   - USB UDC name from `/sys/class/udc/`
2. Generates `device_vars.mk` (Makefile syntax) and `device_vars.env` (shell exports).
3. **`BoardConfig.mk`** includes `device_vars.mk` via `-include` and uses `?=` for fallback values.
4. If no device is found, **`fallback_vars.sh`** runs with hardcoded MT6737M defaults.

## Local Build

To build locally instead of via CI:

```bash
# 1. Initialize TWRP source
mkdir ~/twrp && cd ~/twrp
repo init -u https://github.com/minimal-manifest-twrp/platform_manifest_twrp_aosp.git -b twrp-11
repo sync -j8

# 2. Copy device tree
cp -r device/lge/mmh6lm ~/twrp/device/lge/
cp rootdir/init.rc rootdir/init.recovery.mt6739.rc ~/twrp/rootdir/

# 3. (Optional) Run auto-discovery if device connected
bash scripts/discover_device.sh
cp device_vars.mk ~/twrp/device/lge/mmh6lm/

# 4. Build
cd ~/twrp
source build/envsetup.sh
lunch twrp_mmh6lm-eng
make -j$(nproc) recoveryimage

# 5. Output
ls out/target/product/mmh6lm/recovery.img
```

## Troubleshooting

### UI does not render (buttons invisible)

1. Boot into recovery and check `/tmp/recovery.log`:
   ```bash
   adb shell cat /tmp/recovery.log | grep -i "fb0\|framebuffer\|graphics\|EGL\|surface"
   ```
2. Verify framebuffer permissions:
   ```bash
   adb shell ls -la /dev/graphics/fb0
   # Should be: crw-rw-rw- system graphics
   ```
3. Check `TW_NO_SCREEN_BLANK := true` in `BoardConfig.mk`.
4. If `/sbin/recovery` crashes, check `/tmp/ramdisk-recovery.log` for the crash backtrace.

### ADB does not appear

1. Verify UDC name:
   ```bash
   adb shell ls /sys/class/udc/
   # Should match USB_UDC_NAME in device_vars.mk
   ```
2. Check USB gadget config:
   ```bash
   adb shell ls /config/usb_gadget/g1/configs/b.1/
   # Should show f1 (adb) and f2 (mtp)
   ```
3. Verify `ro.secure`:
   ```bash
   adb shell getprop ro.secure
   # Should be 0
   ```
4. Check `init.recovery.mt6739.rc` — ensure the `rm /config/usb_gadget/g1/configs/b.1/f2` line is **removed**.

### Framebuffer crash

1. Check `/dev/graphics/fb0` exists and has correct permissions (0666).
2. Verify `TARGET_RECOVERY_PIXEL_FORMAT := "RGBX_8888"` in `BoardConfig.mk`.
3. Check kernel log for GPU/display driver errors:
   ```bash
   adb shell dmesg | grep -i "mali\|gpu\|fb\|display\|drm"
   ```
4. Try `TW_USE_FB2_PNG := true` in `BoardConfig.mk`.

## Known Issues

| Issue | Cause | Workaround |
|-------|--------|------------|
| `ro.hardware` mismatch | LG uses `mmh6lm` codename, SoC file is `mt6739.rc` | Hardcoded import in `init.rc` |
| UDC name varies | `musb-hdrc` vs `musb-hdrc.0` vs `dwc3.0` | Dynamic UDC read from `/sys/class/udc/` |
| 32-bit only | MT6737M boots in 32-bit mode (ARMv7-A) | `TARGET_ARCH_VARIANT := armv7-a-neon` |
| No MTP in recovery | Gadget f2 was being removed | `rm f2` line deleted from init.recovery.mt6739.rc |
| Screen blank crash | Some MediaTek panels crash on blank | `TW_NO_SCREEN_BLANK := true` |
